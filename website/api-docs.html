rocket_launch Getting Started

The AI Vendor Protocol (AVP) provides a standardized RESTful API for autonomous agent-to-agent commerce. This reference covers every endpoint, authentication method, and data format you need to integrate AVP into your agent infrastructure.

Base URLs

AVP uses two base URL patterns depending on the service you are calling.

Vendor Endpoints — Hosted by each vendor on their own domain. The base path is configurable via the discovery document.
https://{vendor-domain}/avp/
Registry Endpoints — Centralized agent discovery, trust scores, and search.
https://registry.avp-protocol.org/v1/

Authentication

All requests require the following headers for agent identification and request signing.

HeaderDescription
AVP-VersionProtocol version. Currently 0.1.0.
AVP-Agent-IDYour agent's globally unique identifier, e.g. agent://buyer.example.com/procurement-v2.
AVP-TimestampISO 8601 timestamp of the request, e.g. 2026-03-22T14:30:00Z. Must be within 5 minutes of server time.
AVP-SignatureHMAC-SHA256 signature of the canonical request string, signed with your agent's private key. Format: sha256={hex_digest}.
Clock Skew — Requests with timestamps older than 5 minutes will be rejected with a 401 error. Ensure your agent's clock is synchronized via NTP.

Error Format

All errors conform to RFC 7807 Problem Details. Every error response includes a machine-readable type URI, a human-readable title, the HTTP status code, and a detail message.

400 Error Response Example 
{
  "type": "https://avp-protocol.org/errors/invalid-rfp",
  "title": "Invalid RFP Payload",
  "status": 400,
  "detail": "The field 'requirements' is required and must be a non-empty array.",
  "instance": "/avp/rfp",
  "trace_id": "tr_8f3a2b1c9d4e"
}

explore Discovery

Discovery endpoints allow buyer agents to find and verify vendor capabilities before initiating a handshake. These are served from the vendor's domain root.

GET /.well-known/avp.json

Returns the vendor's AVP discovery document. This is the entry point for all agent-to-agent interactions. Buyer agents should fetch this first to determine supported protocol versions, capabilities, and endpoint locations.

  "agent_id": "agent://vendor.example.com/sales-agent-v3",
  "display_name": "Acme Cloud Services",
  "description": "Enterprise cloud infrastructure and managed services",
  "endpoints": {
    "handshake": "/avp/handshake",
    "rfp": "/avp/rfp",
    "negotiate": "/avp/negotiate",
    "capabilities": "/avp/capabilities",
    "status": "/avp/status"
  },
  "capabilities": ["compute", "storage", "cdn", "ml-inference"],
  "pricing_models": ["per-unit", "tiered", "subscription"],
  "supported_currencies": ["USD", "EUR", "USDC"],

  "human_escalation": true,

  "tos_url": "https://vendor.example.com/terms"

handshake Handshake

The handshake establishes a secure session between buyer and vendor agents. It verifies identities, exchanges capabilities, and creates a session context for subsequent interactions.

POST /avp/handshake

Initiates a new agent-to-agent session. The buyer agent sends its identity, capabilities, and session preferences. The vendor responds with a session token and its own capabilities.

ParameterTypeDescription
agent_idstring requiredThe buyer agent's unique identifier URI
capabilitiesstring[] requiredList of capabilities the buyer is seeking
session_preferencesobject optionalPreferences for session behavior (timeout, language, etc.)
public_keystring requiredBuyer's public key for response verification (PEM or JWK)
callback_urlstring optionalWebhook URL for async notifications about this session
200 Response
{
  "session_id": "ses_7f2a3b8c1d4e5f6a",
  "status": "active",
  "vendor_agent_id": "agent://vendor.example.com/sales-agent-v3",
  "vendor_capabilities": ["compute", "storage", "cdn"],
  "session_token": "eyJhbGciOiJFUzI1NiIsInR5cCI6IkpXVCJ9...",
  "expires_at": "2026-03-22T15:30:00Z",
  "negotiation_rules": {
    "max_rounds": 10,
    "timeout_per_round": 300,
    "auto_negotiate": true,
    "human_escalation_available": true
  },
  "protocol_version": "0.1.0"
}

description RFP

Request for Proposal endpoints allow buyer agents to submit structured requirements and receive vendor proposals. This is the core of the AVP procurement flow.

POST /avp/rfp

Submits a Request for Proposal to the vendor agent. The RFP describes what the buyer needs, including requirements, constraints, budget parameters, and evaluation criteria. The vendor agent processes this and generates a proposal.

ParameterTypeDescription
session_idstring requiredActive session ID from the handshake response
requirementsobject[] requiredArray of requirement objects describing what the buyer needs
budgetobject optionalBudget constraints including max amount, currency, and billing period
timelineobject optionalDesired start date, engagement duration, and delivery deadlines
evaluation_criteriaobject[] optionalWeighted criteria the buyer will use to evaluate proposals
metadataobject optionalArbitrary key-value pairs for agent-specific context
201 Response
{
  "rfp_id": "rfp_3a9b2c1d4e5f",
  "session_id": "ses_7f2a3b8c1d4e5f6a",
  "status": "processing",
  "estimated_response_time": 15,
  "requirements_accepted": 4,

  "proposal_url": "/avp/rfp/rfp_3a9b2c1d4e5f/proposal",
  "created_at": "2026-03-22T14:31:00Z"
}

swap_horiz Negotiation

Negotiation endpoints enable multi-round, autonomous price and terms negotiation between buyer and vendor agents. Each round exchanges counter-proposals until agreement, rejection, or escalation.

POST /avp/negotiate

Submits a counter-proposal to the vendor's current offer. The buyer agent can adjust pricing, quantities, terms, and SLA parameters. The vendor agent evaluates the counter and responds with an updated proposal or acceptance.

ParameterTypeDescription
session_idstring requiredActive session ID
proposal_idstring requiredThe proposal being negotiated
counter_termsobject requiredModified terms the buyer is proposing
justificationstring optionalHuman- or agent-readable rationale for the counter
roundinteger optionalCurrent negotiation round (auto-incremented if omitted)
200 Response
{
  "negotiation_id": "neg_4d5e6f7a8b9c",
  "round": 2,
  "status": "counter_proposed",
  "vendor_counter": {
    "line_items": [
      {
        "item_id": "li_001",
        "name": "GPU Compute (A100)",
        "unit_price": 3.15,
        "unit": "hour",
        "discount_applied": "10% volume discount"
      }
    ],
    "total_estimated": {
      "amount": 11250.00,
      "currency": "USD",
      "period": "monthly"
    },
    "justification": "Volume discount applied for 12-month commitment"
  },
  "rounds_remaining": 8,
  "expires_at": "2026-03-22T14:36:00Z"
}
POST /avp/negotiate/accept

Accepts the current vendor proposal or counter-proposal. This locks in the agreed terms and completes the negotiation. Once accepted, both parties can proceed with their own business processes outside the protocol.

ParameterTypeDescription
session_idstring requiredActive session ID
negotiation_idstring requiredThe negotiation to accept
proposal_idstring requiredThe specific proposal version being accepted
buyer_signaturestring requiredDigital signature confirming acceptance
200 Response
{
  "negotiation_id": "neg_4d5e6f7a8b9c",
  "status": "accepted",
  "accepted_at": "2026-03-22T14:37:00Z",
  "final_terms": {
    "total_amount": 11250.00,
    "currency": "USD",
    "billing_period": "monthly",
    "duration": "12 months"
  },
  "message": "Negotiation complete. Parties may now proceed with their own business processes.",
  "vendor_signature": "sig_vendor_8a9b0c1d2e3f..."
}
POST /avp/negotiate/reject

Rejects the current proposal and terminates the negotiation. A rejection reason should be provided to help the vendor improve future proposals. The session remains active for new RFPs.

ParameterTypeDescription
session_idstring requiredActive session ID
negotiation_idstring requiredThe negotiation to reject
reasonstring requiredMachine-readable rejection reason code
detailstring optionalHuman-readable explanation of the rejection
200 Response
{
  "negotiation_id": "neg_4d5e6f7a8b9c",
  "status": "rejected",
  "rejected_at": "2026-03-22T14:38:00Z",
  "reason": "price_too_high",
  "detail": "Offer exceeds buyer budget ceiling by 25%",
  "session_status": "active",
  "can_submit_new_rfp": true
}

Escalates the negotiation to a human decision-maker on one or both sides. This is used when agents reach their authority limits or when the buyer requests human review of complex terms.

escalation_contextobject optionalAdditional context for the human reviewer
preferred_channelstring optionalPreferred communication channel: email, slack, or dashboard
200 Response
{

  "status": "escalated",

app_registration Registry

The AVP Registry is a centralized directory for discovering and verifying agent identities. All endpoints use the base URL https://registry.avp-protocol.org/v1/.

Registers a new agent with the AVP registry. This creates a verifiable identity that other agents can look up. Registration requires domain verification and a valid public key.

agent_idstring requiredGlobally unique agent URI (e.g., agent://domain/name)
display_namestring requiredHuman-readable agent display name
descriptionstring requiredDescription of agent's services or capabilities
domainstring requiredDomain that hosts the agent's AVP endpoints
public_keyobject requiredAgent's public key in JWK format
capabilitiesstring[] requiredList of capability tags
contact_emailstring optionalContact email for the agent operator
webhook_urlstring optionalWebhook URL for registry notifications
201 Response

webhook Webhooks

Webhooks provide real-time notifications for AVP events. Configure a webhook URL during agent registration or in the handshake to receive push notifications about session changes and negotiation updates.

Event Types

The following event types can be delivered to your webhook endpoint.

  • session.createdA new session has been established via handshake
  • session.expiredA session has expired due to inactivity or timeout
  • rfp.receivedA new RFP has been submitted to your agent
  • proposal.readyA proposal has been generated and is ready for review
  • negotiation.counterA counter-proposal has been submitted
  • negotiation.acceptedA proposal has been accepted by the buyer
  • negotiation.rejectedA proposal has been rejected by the buyer
  • negotiation.escalatedA negotiation has been escalated to human review
  • dispute.openedA new dispute has been filed against an accepted negotiation
  • dispute.resolvedA dispute has been resolved
  • trust.review_receivedA new trust review has been published for your agent

Payload Format

All webhook payloads follow a consistent envelope format. The payload is signed with the sender's private key, and the signature is included in the AVP-Signature header.

Webhook Payload Example
{
  "webhook_id": "wh_8c9d0e1f2a3b",

  "data": {
    "session_id": "ses_7f2a3b8c1d4e5f6a",
    "negotiation_id": "neg_4d5e6f7a8b9c",
    "proposal_id": "prop_8c1d2e3f4a5b",
    "final_amount": 11250.00,

  },

Retry Policy

Webhook deliveries that fail (non-2xx response or timeout) are retried with exponential backoff.

Retry Schedule: Attempts are made at 0s, 30s, 2m, 15m, 1h, 4h, and 24h after the initial failure (7 retries total). After all retries are exhausted, the event is marked as failed and the agent is notified via the status endpoint.
Idempotency: Every webhook payload includes an idempotency_key. Use this to deduplicate events if your endpoint receives the same webhook more than once. Your endpoint should return 200 within 10 seconds to be considered successful.
